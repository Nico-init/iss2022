Introduction

Robot Aspirapolvere

Requirements

Si desidera realizzare un robot aspirapolvere che sia in grado di pulire tutta la stanza in cui si trova
  • [R-1] il Robot deve garantire che tutta la stanza sia pulita, e quindi che passi in tutta la stanza
  • [R-2] deve essere possibile eseguire un test che verifichi la capacita' del Robot di pulire tutta la stanza

Requirement analysis

  • [R-1] il robot deve riuscire a passare su tutta la stanza senza sapere niente dello spazio che lo circonda. Puo' solo muoversi ed accorgersi di essere andato a sbattere
  • [R-2] questo requisito fa intuire che il robot comunichi in qualche modo dove e' passato

Problem analysis

Ci sono vari fattori che incidono sulla capacita' del robot di passare per tutta la stanza:

  • forma della stanza: la stanza e' sempre un rettangolo o puo assumere anche forme piu complesse ed irregolari (magari anche con rientranze)? Soluzioni che vanno bene per alcuni casi potrebbero non funzionare per altri
  • presenza di ostacoli all'interno della stanza: il committente ha poi specificato che la stanza e' sempre vuota
  • la posizione e verso di partenza: per esempio se su una stanza rettangolare il robot parte dall'algolo in alto a sinistra o dal centro potrebbe condizionare la strategia di funzionamento, cosi come se parte girato verso il basso o verso destra.
  • rotazione della stanza
  • dimensioni e la velocita di movimento del robot: essendo che i movimenti del robot sono espressi in millisecondi, lo stesso algoritmo applicato a robot di dimensioni differenti potrebbe non funzionare correttamente

Il problema della verificabilita' del lavoro del robot, comporta che il robot debba comunicare all'esterno dove e' passato. Se si fanno assunzioni come:

  • forma della stanza: rettangolare
  • dimensioni della stanza limitate
  • punto di partenza: in alto a sinistra con robot girato verso basso
potremmo formalizzare un'interfaccia 

public interface testableWorkRobot {
    public boolean[][] cleanRoom();
}

in questo modo se il robot implementa l'interfaccia, sara' possibile verificare il suo operato tramite la matrice di booleani che restituisce. L'idea e' che ogni volta che il robot si muove di MIL millisecondi, setta a true la cella corrispondente. Si potra' poi confrontare tale risultato con quello aspettato

Test plans

Possibile esempio di test Unit.
Prima si setta alla misura desiderata la dimensione della stanza, la posizione del robot, la sua velocita nel file sceneConfig.js 


public class TestRobotCleaner {
private testableWorkRobot robot = new RobotCleaner();

@Test
public void testCleaning(){
  boolean check = true;
  int[][] trueMat = {{true,true,true,true,true,true,true,true,true,true,true,true,true},
              {true,true,true,true,true,true,true,true,true,true,true,true,true}};
  
  int[][] result = robot.cleanRoom();
  
  for(int i = 0; i < trueMat.length && check==true; i++) {
    for (int j = 0; j < trueMat[i].length && check==true; j++) {
      if (trueMat[i][j] != result[i][j]) check = false;  
    }
  }
  assertTrue(check);
}
}

CONSIDERAZIONI: tutta l'analisi credo sia condizionata da una mia idea di soluzione che mi sembra molto semplicistica tipo: "andare avanti fino ad arrivare a sbattere, andare giu, andare avanti fino a sbattere.... fino a che non e' piu' possibile scendere". Da qui vengono le assunzioni che ho fatto nella parte di analisi e la soluzione del test plan, che anche qui mi sembra molto rigida e poco flessibile a variazioni della forma della stanza
